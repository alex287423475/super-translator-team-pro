风格指南

本指南旨在为翻译团队提供一套清晰、可操作的写作规范，确保所有技术文档的中文译文在风格上保持高度一致，并具备卓越的可读性与专业性。请所有译员与审校官在翻译与润色过程中严格遵守。

核心原则

我们的翻译哲学是“优化读者体验”。代码是写给人看的，译文亦然。因此，我们的核心原则是：

清晰至上：一切以提升读者的理解效率为最高准则。避免使用晦涩、冗长或容易产生歧义的表达。
专业一致：保持技术文档应有的专业感，并确保全文术语、语气和格式的绝对统一。
简洁高效：用最精炼的语言传达最准确的信息，避免无意义的修饰词和“翻译腔”。

语言风格与语气

正式而现代：使用标准的现代汉语书面语。避免过于口语化的表达（如“搞定”、“弄一下”），同时也应避免使用文言文或过于古板的句式（如“兹有”、“应予”）。
客观中立：保持客观、中立的陈述语气。避免使用带有强烈个人情感色彩的感叹句或主观评价。
主动语态优先：在中文语境下，优先使用主动语态，这能让句子更清晰、更有力。
示例：
    不佳：该功能将被用来处理数据。
    优选：该功能用于处理数据。
第二人称“你”的使用：在用户指南或教程中，可以直接使用“你”来称呼读者，以增强代入感和指导性。
示例：
    不佳：用户可以通过点击按钮来提交表单。
    优选：你可以点击按钮来提交表单。

术语与用词

强制遵守术语表：所有术语的翻译必须严格遵循 terminology.json 文件中的定义。严禁凭个人喜好或习惯随意翻译。
避免“翻译腔”：警惕英文句式对中文表达的干扰。
长句拆分：英文多长句和从句，而中文多用短句。应将复杂的英文长句拆分为符合中文阅读习惯的几个短句。
示例：
    不佳：这个库，它提供了一个非常强大且易于使用的接口，用于处理各种复杂的网络请求，这对于开发者来说是非常方便的。
    优选：该库提供了一个强大且易用的接口，用于处理各类复杂的网络请求，极大地方便了开发者。
动词的精准使用：选择最精确的动词。例如，“使用”比“利用”更中性，“设置”比“配置”更常用于具体操作，“运行”比“执行”更贴近程序启动的场景。

格式与排版

Markdown 格式：
    加粗：用于强调关键术语或重要概念，但不要滥用。
    示例：在部署之前，请确保已正确设置环境变量。
    行内代码：所有代码相关的词汇，包括变量名、函数名、命令、文件名、API 端点等，都必须使用反引号包裹。
    示例：调用 calculate_average() 函数来计算平均值。
    列表：当列举三个或以上项目时，使用无序列表。
标点符号：
    中英文标点：在中文句子中，请使用全角中文标点（如，。？！：；）。在纯英文的代码、变量名或术语前后，使用半角英文标点。
    示例：如果 status_code 等于 200，则表示请求成功。
    空格：在中文字符与英文单词、数字或代码之间，建议添加一个空格以提升可读性。
    示例：请在 config.json 文件中设置 target_language 参数。

特定内容处理

代码示例：绝对禁止翻译代码围栏（``` ... ```）和行内代码（`...`）中的任何内容，包括变量名、函数名、字符串和注释。它们必须与原文保持 100% 一致。
链接与 URL：禁止翻译 URL 链接本身。如果链接文本（即 [] 中的内容）是可翻译的描述性文字，则可以进行翻译。
示例：
    不佳：访问 https://example.com/docs 获取更多信息。
    优选：访问 获取更多信息。
错误信息与日志：对于程序中可能出现的错误信息或日志输出，如果它们是作为示例出现，应保持原文不变，并在其后用括号附上中文解释。
示例：系统可能会抛出 `ValueError: Input list cannot be empty.`（值错误：输入列表不能为空）异常。
