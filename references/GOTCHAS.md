负面约束与避坑指南

本文件定义了翻译工作流中的“红线”和“禁区”。所有参与工作的智能体（翻译官、审校官、格式工程师）必须严格遵守以下规则。违反这些规则将被视为严重错误。

绝对禁区

以下内容的优先级高于任何翻译策略或风格指南。

代码与变量神圣不可侵犯

禁止翻译代码块：严禁修改代码围栏（``` ... ```）包裹的任何内容。不要翻译其中的注释、字符串或逻辑。
禁止翻译行内代码：严禁翻译反引号（`...`）包裹的内容（如变量名 `user_id`、函数名 `get_data()`）。
禁止翻译占位符：严禁翻译 `{{PH-ID}}` 格式的占位符。它们必须在输出中保持原样，以便后续脚本还原。
禁止翻译文件名与路径：如 config.json、scripts/main.py 等文件名不应被翻译。

格式与结构保护

禁止破坏 Markdown 语法：
    不要删除列表项前的 - 或 *。
    不要破坏表格的 | 分隔符。
    不要随意更改标题的层级（如将 ## 改为 ###）。
禁止破坏 HTML 标签：
    不要翻译 HTML 标签本身（如 `<div>`、`<span>`、`<a>`）。
    不要翻译 HTML 属性名（如 class, id, href）。
    对于 alt="..." 或 title="..." 属性中的文本，除非明确指示，否则通常保持原样或谨慎翻译，不要破坏引号闭合。

链接与引用

禁止修改 URL：严禁更改 http://... 或 https://... 链接本身。
禁止翻译锚点：URL 中的 #section-name 部分必须保留。

翻译陷阱

警惕“假朋友”：
    Event 在编程中通常译为“事件”，但在日历应用中可能是“活动”。请根据上下文判断，但严禁使用错误的通用翻译。
    Library 在编程中是“库”，在图书管理中是“图书馆”。
不要过度本地化：
    保留通用的技术术语英文缩写，如 API, SDK, JSON, HTTP, URL。不要将它们翻译成全称（除非术语表明确要求）。
    不要将 Markdown 翻译成“标记语言”并替换原文。

智能体特定约束

翻译官

禁止遗漏：严禁跳过原文中的任何段落、列表项或注释。即使内容看起来重复或无关紧要，也必须翻译。
禁止臆造：如果原文有模糊不清的地方，请尽量直译并保留原意，不要凭空臆造解释。

审校官

禁止破坏格式：在润色文本时，严禁破坏代码块、行内代码或链接的格式。
禁止过度修改：如果原文翻译已经准确且通顺，不要为了“炫技”而强行修改，导致语义发生微小偏差。

格式工程师

禁止修改内容：你的任务是修复格式，而不是修改文本内容。严禁更改任何非格式相关的文字。
禁止误还原：在还原占位符时，必须确保 `{{PH-ID}}` 与原始内容一一对应，严禁张冠李戴。

错误示例

原文：Call the init() function to start.

错误译文 1：调用 初始化() 函数来开始。（翻译了函数名）
错误译文 2：调用 init() 函数来开始。（保留了英文单词但未用代码格式包裹，导致格式丢失）
错误译文 3：调用 init() 函数来开始。（翻译了 "start" 但语境不佳，应为“启动”）

正确示例

正确译文：调用 `init()` 函数来启动程序。
